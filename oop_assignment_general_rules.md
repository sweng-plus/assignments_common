# General rules:
--------------------------
* Begin your work by carefully reading the text of the assignment as well as the starter code in `src/main/java`, including the comments therein.
* You are expected to:
  * Add code to clases and methods in the starter code, in particular everything marked with `TODO`, and more.
	* Change/add implementation to starter methods. In particular, change dummy return values.
	* Add your own java files, but only in the same package with the given files (pay attention to the package declarations).
* You may add to starter classes: new attributes and methods, inheritance (super- or subclasses), and inner classes (if really needed).
* Regarding the starter code, please make sure **not** to:
	* rename the root package (folder that resides in `src/main/java`)
    * rename classes and their access modifiers 
    * change method signatures
    * modify the contents of the `.github` folder and `pom.xml` file.
    * modify existing test classes in the folder src/test/java (if any)
* **Important: if at least one of the above rules is not satisfied, one or all tests may fail and points may be lost!**


# Regarding testing
------------------------------
* You should always include a main method (also class, if needed) with minimal testing of your code.
* Immediately after the deadline, a testing workflow is executed on your software and a number of points is calculated based on the testing results.
* You can run the testing workflow on Github for a given (small) number of times before the deadline.
* The testing workflow can also be run locally as many times as you want if you work with git and maven.
* It is normal at the beginning to have compiler errors in the starter code, or otherwise failed tests, since the starter code does not satisfy the requirements of the assignment. 


## How to run tests on Github
------------------------------
* On the repository Github page, select Actions(tab on top) -> Github Classroom Workflow (tab on the left). There you'll see a list of all runs of this workflow so far. On top of that list to the right there is a button "Run workflow".
* Click on that, and in the pop-up window click again on the green button with the same name. 
* Wait for the page to be updated (may take several minutes). 
* You will get a summary page with information about how many runs are still available to you and the results of compilation and testing, as well as the resulting number of points.


# After submission deadline
------------------------------
* One last run of the testing workflow will be executed automatically.
* The final number of points  will be shown on the badge in the readme (visible on the repo page). They will be later transferred to Blackboard.
* You won't have writing access to your assignment repository anymore, so you won't be able to push changes after the deadline.
* However, you will still be able to read the repository until the end of the semester.
* The repository will be deleted after semester end.
